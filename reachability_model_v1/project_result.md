# 都市道路ネットワーク分析・新規道路建設最適化プロジェクト

## プロジェクト概要

**タイトル:** 都市道路ネットワーク構造の定量分析と、コスト最小・効果最大を目標とした新規道路建設地点の最適化

**目的:** OpenStreetMap（OSM）の道路データと `osmnx` ライブラリを活用し、任意の都市における「道路ネットワークの交通効率」を定量化する。さらに、交通効率が著しく低い（分断されている）地点を特定し、**最も短い道路・橋を最も少ない投資で建設することで、最大の効率改善をもたらす最適な建設場所**を自動的に提案するアルゴリズムを開発する。

**対象都市（本実験）:** 金沢市（石川県）
- 中心部に密集した城下町の街区（高効率エリア）
- 犀川・浅野川という2本の川（物理的バリア）
- 東部から南部にかけての山間部（低効率エリア）
という対照的な特徴を併せ持つ、中規模地方都市として最適なテストベッド。

## Phase 1: 到達圏（IsoDistance）の面積算出手法の確立

### 1-1. 問題設定：「道路ネットワーク（線）から面を作る」難しさ

都市の「交通効率」を定量化するには、「ある地点から一定距離内に到達できる面積（IsoDistance Area）」と、「障害物が何もない場合の理論上の到達面積（ユークリッド円：$\pi r^2$）」を比較することが有効である。この比率 **IoU (Intersection over Union)** が高いほど、その地点から「本来到達できるはずの広さ」を実現できていることを意味する。

$$\text{IoU} = \frac{\text{実際の到達面積}}{\pi r^2}$$

しかし、道路データはノード（交差点）とエッジ（道）の「線」として存在し、これを「面（ポリゴン）」に変換するアルゴリズムが問題の核心となる。

---

### 1-2. 検討した手法と問題点

#### 手法A: Convex Hull（凸包）

全ての到達ノードを囲む最小の凸多角形を生成する手法。

**実験:** 京都市 烏丸御池周辺（ネットワーク距離1.5km圏内）

![Polygon Evaluation Plot](/Users/pontarousu/.gemini/antigravity/brain/6e76f7ef-41b2-415c-92db-2f694ba76e8e/polygon_evaluation_plot.png)

| 評価点 | 内容 |
|--------|------|
| ✅ シンプルで高速 | 計算が軽量 |
| ❌ 致命的欠陥 | 川・皇居・山など「到達不能な空白」を面積に含む「嘘のポリゴン」が生じる |

**東京駅での実証:** 皇居の堀（絶対に通れない）を跨いでポリゴン化してしまう。

![Mesh Evaluation Plot](/Users/pontarousu/.gemini/antigravity/brain/6e76f7ef-41b2-415c-92db-2f694ba76e8e/mesh_evaluation_tokyo.png)

#### 手法B: Alpha Shape（凹包）

「丸みの半径（αパラメータ）」を設定して凹んだ輪郭を描く手法。

| 評価点 | 内容 |
|--------|------|
| ✅ Convex Hullより輪郭が現実的 | |
| ❌ 不安定 | αパラメータの値によって結果が大きく変化し、都市ごとに調整が必要 |
| ❌ 依然として障害物を跨ぐ | 内部バリア（川・池）を貫通してしまう |

#### 手法C: Edge Buffer（道路バッファ）

道路エッジを一定幅（例: 20m）で太らせてポリゴン化する手法。

| 評価点 | 内容 |
|--------|------|
| ✅ 道路に沿って正確 | 到達不能領域を含まない |
| ❌ 面積が不連続 | 街区（ブロック）の内部に穴が空き、面として意味をなさない |

#### 手法D: Grid Mesh（格子メッシュ法）

空間を一定サイズ（50m）のグリッドに分割し、道路エッジと交差するセルのみをカウントする方法。

| 評価点 | 内容 |
|--------|------|
| ✅ 皇居問題を解決 | 道路がないエリアはグリッドが塗られない |
| ❌ 山間部で破綻 | 山間部の1本道では面ではなく細い線にしかならない |
| ❌ メッシュサイズのジレンマ | 細かくしすぎると手法Cと同じ問題、粗すぎると精度が下がる |

> **メッシュサイズのパラドックス:** メッシュを極限まで細かくすると Edge Buffer と等価になり、逆に粗くするとConvex Hullに近づいてしまう。「最適なメッシュサイズ」は街区のブロックサイズ（50〜100m）に依存するため、街ごとに異なる。

### 1-3. ✅ 採用手法：コスト・ラスタ歩行シミュレーション（Cost Raster / Fast Marching Method）

#### 理論的背景（アイコナール方程式）

GISの到達圏解析で使われるプロフェッショナルな手法。空間全体を細かいピクセルに分け、各ピクセルに「移動コスト（摩擦係数）」を定義する。

$$||\nabla T(x,y)|| = C(x,y)$$

これはアイコナール方程式（光・波の伝播を記述する物理方程式）の離散化に相当し、Fast Marching Method（高速前進法）またはダイクストラ法で解かれる。

$$T(p) = \min_{q \in N(p)} \left(T(q) + C(q \to p) \cdot ||p-q||\right)$$

#### コスト設定

| ピクセルの種類 | 移動コスト | 解釈 |
|--------------|-----------|------|
| 道路エッジ上 | **1** | 車道・歩道を高速に移動 |
| 一般空間（家屋・公園等） | **20** | 徒歩での侵入（20倍の時間） |
| 水域・障害物（川・堀等） | **∞** | 絶対に通過不可能 |

#### 3都市での検証実験

![Cost Raster Evaluation](/Users/pontarousu/.gemini/antigravity/brain/6e76f7ef-41b2-415c-92db-2f694ba76e8e/cost_raster_evaluation.png)

| ケース | 結果 |
|--------|------|
| **京都（碁盤の目・都市部）** | ブロック内部まで波紋が染み渡り、面として完全に塗りつぶされた |
| **東京（皇居あり・障害物都市部）** | 堀（水域・コスト∞）で波面が完全停止。「嘘のポリゴン」が一切生じない |
| **箱根（山間部）** | 道路から数十mのリボン状の面積が正確に算出。山全体を覆うような誤りがない |

**結論:** 都市・山間・障害物ありの3ケース全てで物理的に矛盾のない到達面を表現できた。**以降の全ての分析でこの手法をIoU計算のベースラインとして採用する。**

## Phase 3: 新規道路建設の最適化（ROI最大化）

### 3-1. 問題設定と目標関数

Phase 2で特定した「ボトルネック（IoUが低い地点）」に対して、最小コストで最大効果をもたらす新規道路（橋）の建設場所を探す最適化問題として定式化する。

#### 目標関数（目的関数）

$$\text{ROI} = \frac{\sum_{i \in \text{影響圏}} \Delta\text{IoU}_i}{L_{\text{edge [m]}}}$$

- **分子（効果）:** 追加した仮想道路の影響圏（半径1500m）内にある全サンプル地点での「IoU改善量の合計」
- **分母（コスト）:** 追加する道路（橋）の長さ（メートル）
- **解釈:** 「道路1mあたりに、周辺何地点のIoUをどれだけ改善できるか」の指標。**高いほどコストパフォーマンスに優れた建設地点**。

#### なぜ新設道路の周辺「以外」のIoUも改善するか：ネットワーク効果

新設道路上に存在しないサンプル点のIoUが改善する現象は、**ネットワーク効果（Network Effect）**によるものであり、バグではない。

コスト・ラスタ上を水のように広がる波面は、**新設道路（低コスト回廊）を「経由地点」として利用**できるようになる。橋の真上に住んでいなくても、橋まで歩ける距離なら橋を渡った先の対岸にも到達できるようになる。これは現実の交通インフラが持つ「一点の改善が広域を恩恵する」性質を数値・空間的に正確に再現している。

```
【BEFORE】サンプル点Aの到達圏
  ~~川（コスト∞）~~
A● → → → |×|  ← ブロック、到達不可

【AFTER】新しい橋を追加した場合
  ~~川（コスト∞）~~
A● → → → [橋] → → → ●  ← 対岸まで到達！
```

---

### 3-2. アルゴリズムの手順（詳細）

#### 全体の考え方：「物理的に近いのに、道路上は遠い場所」を探す

新しい道路が最も「効く」のは、以下のような場所です：

```
地点 A と 地点 B が...

  直線距離（Euclidean）: 150m（目と鼻の先）
  　↕ 間に川がある
  ネットワーク距離: 1,200m（橋を探して大回り）

→ ここに橋を1本架けるだけで、1,200m → 150m に短縮できる！
```

「空間的には近いのに、道路ネットワーク上では遠回りを強いられているノードペア」を検出することが候補探索の目的です。

---

#### Step 2: 候補ノードプールの絞り込み（設計上の最適化）

金沢市全体のグラフには約**6,057ノード（交差点・端点）**があります。全ペアを総当たりすると：

$$\binom{6057}{2} \approx 18,000,000 \text{ 組}$$

これは現実的ではありません。そこで計算量を削減するため、「ボトルネック地点群の重心座標」を中心に一定半径内（`DIST × 0.8 = 4km`）のノードのみに絞り込む処理を設計しています。

**設計意図：** ボトルネックが都市の特定の区域（例えば北東部）に集中している場合、南西部の無関係なノードを候補から除外することで計算量を大幅に削減できます。

**今回の実情（金沢市の場合）：**
しかし今回は、ダウンロードした道路データの範囲がちょうど中心から5km圏内に収まっており、ボトルネック108地点の重心も金沢市中心部付近に位置したため、**全6,057ノードが絞り込み半径4km内に含まれてしまいました。** 結果として「フィルタを適用したが、実質的に何も除外されなかった」状態です。

> ➡️ このフィルタは都市スケールが大きい場合（例: 東京23区全体など）に初めて有効に機能します。今回の実験では意味を持ちませんでした。

---

#### Step 3: ランダムサンプリング（300ノード）

絞り込んだプールからさらにランダムに**300ノード**を選びます。この300ノード間の全ペア数は：

$$\binom{300}{2} = 44,850 \text{ 組}$$

この範囲であれば次のStep（ネットワーク距離の計算）が現実的な時間で終わります。

> ⚠️ **ランダムサンプリングの限界（今後の課題）：** ランダムにサンプリングしているため、本当の最適候補が選ばれないノードに含まれる場合があります。空間インデックス（KD-Tree等）を用いた全数探索への拡張で精度改善が見込まれます。

---

#### Step 4: 2条件フィルタによる候補絞り込み（最重要ステップ）

各ノードペア $(A, B)$ に対して、以下の **2つの条件を両方満たすもの** だけを候補とします。

**条件 ①：空間的近接（Euclidean距離 ≤ 300m）**

$$d_{\text{Eucl}}(A, B) \leq 300\text{m}$$

理由：建設する道路（橋）が300m以内であれば、現実的なコストで実現できる。これより遠いと「建設コスト（長さ）」が大きすぎてROIが下がる。

**条件 ②：迂回の大きさ（ネットワーク迂回比 ≥ 3.0倍）**

$$\frac{d_{\text{network}}(A, B)}{d_{\text{Eucl}}(A, B)} \geq 3.0$$

理由：現状の道路ネットワーク上での移動距離が、直線距離の3倍以上あるペアは、「バリア（川・線路等）によって大迂回を強いられている」と判断できる。ここに道路を追加すると劇的な短縮効果が見込まれる。

**2つの条件を組み合わせる理由：**

| 条件 | 単独使用の問題点 |
|------|----------------|
| ①のみ | 既にスムーズに繋がっているペアも大量に入る（効果の薄い候補が増える） |
| ②のみ | 何km先のペアも候補になり「10kmの橋を建設」等の非現実的提案が生まれる |
| **①＋②を組み合わせ** | **「短くて、かつ迂回を解消する効果が大きい」という条件を同時に満たす** |

**具体例で確認：**

```
ペア (A, B):
  dEucl = 150m
  dNetwork = 1,200m  → 迂回比 = 8.0倍 ✅ 両条件クリア → 候補

ペア (C, D):
  dEucl = 200m
  dNetwork = 250m   → 迂回比 = 1.25倍 ✗ ②を満たさない → 候補外
  （既に近くに道があるため、新道を作っても効果が薄い）

ペア (E, F):
  dEucl = 500m
  dNetwork = 3,000m  → 迂回比 = 6.0倍 ✗ ①を満たさない → 候補外
  （効果は大きそうだが、500mの橋は建設コストが高すぎる）
```

**今回の結果:** このフィルタを通過した候補は **19組** でした。

---

#### Step 5: 各候補のROI評価

条件をくぐり抜けた19候補のそれぞれについて、以下の処理を行います：

```
1. コストラスタに仮想エッジ（コスト=1の細線）を焼き込む
   （「仮想的にその場所に道路が存在する」と設定する）

2. 提案道路の中心点から半径1500m以内のサンプル地点を「影響圏」として特定

3. 影響圏内の各地点でIoUを再計算
   （仮想道路ありのコストラスタで、Fast Marchingを再実行）

4. ΔIoU = 新IoU - 元IoU（改善量）を各地点で算出

5. ROI = sum(ΔIoU) / エッジの長さ(m)
```

この計算を19候補全てに対して並列実行し、ROI降順でランキングを出力します。

## 成果のまとめと考察

### プロジェクトの3大成果

1. **普遍的な到達面積算出手法の確立**
   コスト・ラスタ法（Fast Marching Method）が、都市部・山間部・障害物ありの全ての地形において物理的に整合した到達圏面積を算出できることを実証した。

2. **金沢市全域の交通効率ヒートマップの生成**
   並列処理を活用し、361箇所のサンプル点を23秒で計算。川や山脈による「見えない交通バリア」を数値的・視覚的に明示した。

3. **最小コスト・最大効果の新規道路建設点の特定**
   ROI = ΔIoU合計 / 道路長 という目標関数を定義し、金沢市内で最も投資対効果の高い建設候補地（Rank1: 116m、迂回9.6倍解消）を定量的に特定した。

## 今後の展望

| 課題 | 内容 |
|------|------|
| **候補探索の全探索化** | 現在はランダムサンプリング（300ノード）。空間インデックス（KD-Tree等）を用いた全ノードペアの網羅的探索で精度向上が見込める |
| **逐次最適化（Greedy拡張）** | 1本追加→再計算→次の1本、という手順を繰り返し「建設予算配分計画・工事順序」を提案するアルゴリズムへの拡張 |
| **複数都市への横断適用** | 高知市・富山市・長崎市など他の中規模地方都市に適用し「都市の構造的特徴スコア」を比較分析する |
| **DEM（標高データ）の統合** | 山間部での徒歩コストに勾配（坂の傾き）を加味し、より現実的な摩擦係数の設定を実現する |
| **道路種別の重み付け** | 国道・県道・市道・細道などの道路種別ごとに異なる通行速度コストを設定し、精度を向上させる |
