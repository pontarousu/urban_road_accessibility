# 理論検討ログ (Theory Discussion)

## 1. ポテンシャル場からグラフ理論への転換 (Step 1)
当初は流体力学・ラスタ表現（画像ピクセル上のナビエ・ストークス方程式やポアソン方程式）を想定していた。しかし、壁(障害物)の計算負荷や「道幅」という現実のインフラ制約を表現できない課題が浮上。ユーザーの「道に線を引く形にしたい」という直感に基づき、Cell Transmission Model (メゾスコピック交通流モデル) と Dijkstra探索への転換が行われた。この直感は計算効率と理論的双方が極めて優れていた。

## 2. 経路分岐(パケット分裂)の物理法則 (Step 2)
### True Fork Node問題
OpenStreetMap上の交差点は、物理的には1つの曲がり角でも、データ上は微小なエッジで構成された複数ノード群になることが多い。これを解決するため、経路配列の共通要素を突合し、「本当に道が分かれる最初のノード」のみを分岐点として扱うアルゴリズムを発見した。
### アニメーションにおける相対速度の落とし穴
フレーム数を固定すると長い道では超高速に、短い道では超低速になる。物理モデルとして「距離を速度で割ってフレーム数を逆算する」という現実の物理学に基づいた補間（`LineString.interpolate`）を実装した。

## 3. 渋滞アルゴリズムと「動的・中途経路変更（User Equilibriumの萌芽）」 (Step 3 & 3.5)
### BPR関数による All-or-Nothing のジレンマ (Step 3)
初期の渋滞シミュレーションでは、先客の大容量パケットが「出発時に決めた最短ルート」をバカ正直に最後まで突き進むため、一本道で大渋滞を引き起こした。後続パケットはそれを賢く迂回した。
### ユーザーからの究極の指摘 (Co-Evolution Insight)
**「渋滞が発生したら、その赤ノードにいるユーザー自身も分裂して別ルートを探すのではないか？」**
これは交通工学における**「利用者均衡（Wardrop's User Equilibrium）」**の概念そのものである。
「パケットが出発時に決めたルートを盲信する」のではなく、「移動中に目の前の道が混み始めたら、パケット内の一部（賢いドライバー達）が離脱して新しい最短ルート（迂回路）へと流れ出し、結果的にパケットが自己増殖的に分裂していく」という動的な挙動への昇華が提唱された。
これにより、Step 2（人為的な宛先違いでの分裂）とStep 3（道幅オーバーによる渋滞）が融合し、「渋滞による自発的な分裂（迂回）」という極めて高度な複雑系モデルへの進化が決定づけられた。

### 「パケットか、個人か」— ミクロとメゾスコピックの本質的な違い (Co-Evolution Insight)
ユーザーからの重要な指摘:
**「1500人のパケットが意思を持って分裂しているのではなく、あくまで1500人の独立した個人が一人一人ルートを選択している。ただし全員を個別計算（ミクロシミュレーション）すると計算量が爆発するから、マクロ的な塊（パケット）として近似処理している」**

この指摘はメゾスコピックモデルの基盤思想そのものであり、以後のコード実装と表現の両方に反映すべき根幹原理として確立された。

## 4. 逐次シミュレーション・エンジンへの転換 (Step 3.5 再設計)

### 旧実装（`evaluate_dynamic_equilibrium_split.py`）の問題点分析
AIモデル変更後のコードレビューにより、以下の5つの構造的問題が特定された。

1. **分裂地点がハードコード**: `route_origin[len//4]`（ルートの25%地点）を機械的にボトルネックとして指定しており、Capacityに基づく物理的判定が存在しなかった。
2. **迂回路生成がダミー注入に依存**: Volume=5000という根拠のない値を注入し「とりあえず別の道が出ればOK」という結果ありきの設計だった。
3. **TRAFFIC_Aの決定が恣意的**: `min(edge_cap, 500)` のように「500」という上限に理論的根拠がなかった。
4. **逐次ステップ（ループ）が存在しない**: 全ルートを事前計算し、見た目だけのアニメーションを再生する「映像装置」だった。交差点ごとに判定するシミュレーションが存在しなかった。
5. **Step 2のTrue Fork Node設計との断絶**: Step 2で学んだ「道が共通している間は1つのパケットとして扱う」という教訓が活かされていなかった。

### 新アーキテクチャ: 逐次シミュレーション・エンジン (`simulate_equilibrium.py`)

**設計原理**: 事前に「どこで分裂するか」を決めるのではなく、パケットが交差点に到着するたびに `packet.size > edge.capacity` を評価し、超過時にのみ分裂を発生させるイベント駆動型ループ。

**アルゴリズム（1ステップの処理）**:
1. 各パケットが現在位置の次のエッジのCapacityを確認
2. `size <= capacity` → 全員通過、1ホップ前進
3. `size > capacity` → 分裂:
   - グループA（capacity人）: そのまま直進
   - グループB（残り）: 直進エッジを「完全渋滞」としてBPR重みを再計算し、Dijkstraで迂回路を再探索
4. BPR重みをグラフ全体で更新
5. 全パケット到着まで繰り返し

**初回実行結果（2026-04-14）**:
- 1500人 → Node 1590192773 で4回連続分裂 → 300人×5パケットに自己組織化
- 32ステップで全パケットがゴールに到達
- 同一ノードで連続分裂するケースは、迂回路の最初のエッジが同じ交差点を経由するために発生する（意図通りの挙動）

## 5. 物理モデルの修正（v2: 2026-04-14）

### 問題1: Volume の単調増加（退出処理の欠落）
パケットがエッジに進入した際に volume が加算されるが、次のエッジに移動した際に減算されていなかった。
結果としてシミュレーション後半では、誰も走っていないはずの道路のBPRコストが異常に高いままとなり、
不当に遠回りを強いられる状態が発生。

**修正**: `recompute_volumes_from_packets()` を各ステップ冒頭で呼び、パケットの `occupied_edge` から
volume を毎回ゼロベースで再計算する方式に変更。累積ドリフトを構造的に排除。

### 問題2: 次元の不整合（絶対数 vs フローレート）
分裂判定 `N > C_e` において、N は「台（絶対数）」、C_e は「台/h（フローレート）」であり、
次元が一致していなかった。

**修正**: 「リンク貯留容量 (Link Storage Capacity)」を導入。
$$C_e^{\text{storage}} = \frac{\ell_e}{7.5\text{m}} \times n_{\text{lanes}}$$
これは「この道に物理的に何台が同時に入れるか」を表す絶対数であり、
パケット人数 N との比較が次元的に正しくなる。

## 6. 段階的投入の導入（v3: 2026-04-14）

### ユーザーからの指摘 (Co-Evolution Insight)
**「スタート地点に1500台が同時にいるのは非現実的。実際の道路では、ある程度の時間をかけてその台数が来る。」**

この指摘は交通工学における「需要の時間分布（Demand Profile）」の概念に対応する。
ピーク時間帯の交通量は一瞬で発生するのではなく、時間をかけて累積する。

**修正**: `injection_rate` パラメータを導入。1ステップあたり100台ずつ、15ステップかけて
1500台を段階的にSTARTノードへ投入。先行バッチが道を空けてから後続バッチが到着するため、
貯留容量の小さい住宅地道路でも自然に交通が流れるようになった。

**v3 実行結果（2026-04-14）**:
- 100人/step × 15ステップ = 1500人を段階的投入
- 65ステップで全パケットがゴールに到達
- 最終パケット数: 1000（min=1人, max=4人, median=1人）
- 分裂はネットワーク全体に分散し、複数のボトルネック交差点で自然に発生
